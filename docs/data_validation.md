# Data Validation

The validation package checks generated and persisted AML datasets before they are transformed into
staging tables. It validates local pandas DataFrames and persisted reference dataset metadata, then
can write local failure artefacts for review and lineage.

## Table Schemas

Pandera schemas are defined for:

| Table | Purpose |
| --- | --- |
| `countries` | Country reference data and jurisdiction risk scores |
| `customers` | Customer identity, segment, jurisdiction, and risk fields |
| `accounts` | Customer-owned accounts and account lifecycle fields |
| `counterparties` | External parties, institutions, and risk scores |
| `devices` | Digital device identifiers used by transactions |
| `transactions` | Baseline and scenario transaction activity |
| `scenario_manifest` | Optional AML scenario evidence metadata |

## Required Columns

The schemas enforce the core columns generated by the reference package:

- countries: `country_code`, `country_name`, `region`, `is_high_risk`, `risk_score`
- customers: `customer_id`, `customer_type`, `customer_segment`, `jurisdiction`,
  `occupation`, `industry_code`, `customer_risk_rating`, `customer_risk_score`,
  `onboarded_at`
- accounts: `account_id`, `customer_id`, `account_type`, `account_status`, `currency`,
  `opened_at`, `closed_at`, `home_country`
- counterparties: `counterparty_id`, `counterparty_type`, `counterparty_name`, `country_code`,
  `institution_name`, `external_account_ref`, `risk_score`
- devices: `device_id`, `device_type`, `ip_address`, `ip_cluster`, `phone_hash`,
  `browser_fingerprint`
- transactions: `transaction_id`, `sender_account_id`, `receiver_account_id`,
  `counterparty_id`, `device_id`, `transaction_timestamp`, `amount`, `currency`,
  `transaction_type`, `channel`, `origin_country`, `destination_country`, `is_cross_border`,
  `is_labelled_suspicious`, `typology_label`, `source_file`, `scenario_id`, `scenario_role`,
  `scenario_sequence`, `scenario_injected`

## Transaction Checks

Transaction validation enforces unique non-null transaction IDs, populated sender accounts,
parseable timestamps, strictly positive amounts, supported transaction types and channels, populated
origin and destination countries, boolean-like flags, and required typology labels for suspicious
transactions.

When `scenario_injected=True`, `scenario_id` must be populated so scenario evidence can be linked
back to the manifest.

## Scenario Checks

The optional `scenario_manifest` schema validates scenario IDs, typology labels, primary accounts,
evidence transaction references, detection windows, expected rule names, expected severities, and
reason codes. CSV round-tripped list-like fields are accepted as non-empty strings.

## Referential Integrity

Cross-table checks validate that:

- customer jurisdictions exist in countries
- account customer IDs exist in customers
- account home countries exist in countries
- counterparty country codes exist in countries
- transaction sender accounts exist in accounts
- transaction receiver accounts exist when populated
- transaction counterparties exist when populated
- transaction devices exist when populated
- transaction origin and destination countries exist
- `is_cross_border` matches `origin_country != destination_country`
- manifest evidence transactions exist in transactions
- manifest account references exist in accounts
- transaction scenario IDs exist in the manifest when present

## Validation Results

Validation returns immutable result objects:

- `TableValidationResult` records pass/fail status, row count, error count, and table-level errors.
- `DatasetValidationResult` groups table results and referential errors.

The summary helper returns counts for passed tables, failed tables, total rows, total errors, and
referential errors. Markdown output is intended for later validation reports.

## Validation Failure Handling

Handled validation creates a validation run record and, by default, writes local artefacts under
`reports/validation_failures`. These outputs are designed for data quality review before raw
ingestion or later governance reporting.

The high-level workflow does not write audit events directly and does not connect to PostgreSQL.
Audit writing is an explicit CLI option.

Staging transformations reuse validation checks before loading clean relational tables. Because the
current staging transaction table does not include labelled scenario metadata columns, staging
validation adds those columns only to a transient validation copy.

## Validation Run Metadata

Each handled run receives a readable validation run ID containing the dataset ID, dataset version,
and a UTC timestamp. `validation_run_metadata.json` records:

- dataset ID, dataset version, and dataset type
- metadata path
- status: `passed`, `failed`, or `failed_strict`
- strict mode flag
- table count, total rows, failed table count, total errors, and referential error count
- paths to generated artefacts when available

## Invalid Record Artefacts

Pandera failure cases are used to extract invalid source rows where row indexes are available.
Invalid record CSVs include:

- `validation_table`
- `validation_error`
- `validation_check`
- `validation_column`

When a failure cannot be tied to a specific input row, the artefact contains a compact failure frame
with enough schema failure context to review the issue.

## Referential Error Artefacts

Cross-table errors are written separately to `referential_errors.csv`. This keeps row-level schema
failures distinct from relationship and integrity failures such as unknown account, device,
counterparty, country, or scenario references.

## Validation Audit Events

`--write-audit` inserts one row into `governance.audit_events` with:

- `event_type`: `data_validation`
- `component`: `validation`
- `action`: `validate_persisted_dataset`
- `status`: validation run status

The audit event details JSON includes validation run ID, dataset identity, strict flag, validation
counts, metadata path, and artefact paths. Audit writing requires PostgreSQL schemas and tables to
exist.

## Strict Mode

Strict mode marks failed handled runs as `failed_strict` and makes the CLI exit `1` when validation
fails. Without strict mode, failed validation writes artefacts and exits `0` so local diagnostics can
continue.

## Artefact Directory Layout

Example output layout:

```text
reports/validation_failures/
|-- synthetic_scenario_seed_42_scenario_42/
|   |-- 20250101T120000Z/
|       |-- validation_<timestamp>/
|           |-- validation_summary.json
|           |-- validation_report.md
|           |-- validation_run_metadata.json
|           |-- invalid_records/
|           |   |-- invalid_transactions.csv
|           |-- referential_errors.csv
```

## CLI Usage

Validate an explicit persisted dataset:

```bash
python scripts/validate_data.py validate --metadata-path data/.../dataset_metadata.json
```

Validate the latest scenario dataset:

```bash
python scripts/validate_data.py validate-latest --dataset-id synthetic_scenario_seed_42_scenario_42
```

Strict mode exits with code `1` on validation failure:

```bash
python scripts/validate_data.py validate-latest --dataset-id synthetic_scenario_seed_42_scenario_42 --strict
```

Without `--strict`, validation failures are printed but the command exits `0` for exploratory local
diagnostics.

Disable artefact writing:

```bash
python scripts/validate_data.py validate-latest --dataset-id synthetic_scenario_seed_42_scenario_42 --no-artefacts
```

Write validation artefacts to a custom base directory:

```bash
python scripts/validate_data.py validate-latest --dataset-id synthetic_scenario_seed_42_scenario_42 --output-dir reports/validation_failures
```

Write an audit event:

```bash
python scripts/validate_data.py validate-latest --dataset-id synthetic_scenario_seed_42_scenario_42 --write-audit
```

## Makefile Usage

```bash
make generate-data-scenarios-small

make validate-data-audit
```

Baseline validation:

```bash
make generate-data-small
make validate-data-baseline
```

## Limitations

This ticket validates in-memory and persisted local files, writes validation failure artefacts, and
can write an explicit validation audit event. It does not transform raw data into staging tables,
execute AML rules, construct graphs, train models, score risk, or generate cases.
